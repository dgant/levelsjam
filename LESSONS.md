# Lessons Learned

- React Three Fiber scene content must be rendered inside a `<Canvas>` host. A production build can succeed while the runtime still fails if hooks like `useFrame` and `useThree` are mounted outside the canvas context.
- The Vite `base` setting should stay relative for this project unless the repository name is fixed. `base: './'` keeps GitHub Pages assets working under either `/levelsjam/` or a future `/jam2026/` path.
- This repository's current GitHub Pages token cannot switch Pages to workflow mode. The reliable publish path is legacy `main` root hosting with committed built assets at the repository root.
- Runtime-served textures must live under `public/` in source. The root-level `textures/` directory is generated Pages output and should be refreshed directly from `public/textures` during `npm run build:pages` instead of being round-tripped through `dist`.
- When movement tuning is specified in both speed and distance terms, preserve both constraints in the spec so tests can validate the intended feel instead of a single derived number.
- When agent workflows need live Vite access, prefer one persistent headless instance over repeatedly starting visible windows.
- If the root `index.html` chooses between source and committed bundle modes, include every approved Vite dev port in that detection. Otherwise a background server can silently serve stale built assets.
- The April 3, 2026 startup benchmark showed that the scene reaches its first rendered frame in about 1-2 seconds on the dev server, while the slowest remaining startup requests are remote `@takram/three-atmosphere` EXR files from `media.githubusercontent.com`.
- Playwright should only match `*.spec.cjs` in this repository. If it scans the Node `--test` files, it wastes time running them inside the browser-test command.
- For rendered-frame smoke checks in this scene, broad brightness sampling across the whole canvas is more reliable than checking a single center pixel.
- In this project, moving the `three-atmosphere` EXRs into `public/textures/atmosphere` cuts startup latency dramatically compared with loading them from `media.githubusercontent.com`.
- The `canvas[data-scene-ready]` marker proved less reliable than the rendered image for startup benchmarking, so the benchmark now waits for a bright canvas frame instead of a custom dataset flag.
- After Playwright diagnostics or benchmarks, explicitly stop the headless browser processes they spawned instead of assuming they will all exit cleanly on their own.
- `@react-three/postprocessing` forces the renderer to `NoToneMapping` while the composer renders, so visible tone-mapper controls must drive a `ToneMapping` post effect rather than `gl.toneMapping`.
- The custom lens flare shader from `@react-three/postprocessing` treats its `opacity` uniform as suppression, not visibility, so app-side code must invert that value or the effect stays hidden.
- In this project, `@takram/three-atmosphere` light-source lighting behaves coherently only when the visible sky, `SunLight`, `SkyLight`, water sun color, and any specular environment map all derive from the same atmosphere state instead of mixing in manual white sun colors.
- The visual `sun intensity` control should be normalized around the package's default light strength. Passing the raw UI value straight into `SunLight`, `SkyLight`, or water sun color exaggerates highlights and breaks the sky-to-sun balance.
- For this project, the cleanest physically anchored workflow is to expose direct sun illuminance in lux and camera exposure in EV100 at the UI layer, then convert those inputs into stable internal scales for `SunLight`, `SkyLight`, atmosphere solar irradiance, PMREM capture, and renderer exposure.
- `three-atmosphere` can keep plausible atmospheric color while still fitting half-float precision, so the right compromise here is explicit lux and EV100 calibration rather than trying to force every internal buffer into strict photometric units.
- In headless Playwright for this project, clipped page screenshots of the canvas are more reliable than sampling the WebGL canvas via `drawImage` or `toDataURL`, and they still fit the smoke-test budget when the viewport and sample region stay small.
- The smoke benchmark is highly sensitive to Playwright screenshot work and viewport size. Keeping the viewport moderate and avoiding extra image-diff assertions preserves the under-one-minute runner budget.
- Older SSR libraries in the three.js ecosystem can lag behind current shader-chunk and MRT APIs. In this project, treat SSR as an optional debug toggle and verify it against the exact installed three.js version before enabling it by default.
- The local HDRI plus ShareTextures wall scene still benefits from `canvas[data-scene-ready]`, but smoke tests should not assume that dataset flips immediately after the loading overlay disappears.
- Playwright's `webServer` teardown can lag briefly after a smoke run. In this repository, polling `127.0.0.1:42731` until it is free makes back-to-back smoke benchmarks reliable without re-enabling server reuse.
- After a Windows crash or forced reboot, `.vite-dev-server.json` can contain garbage bytes. The dev-server helper should treat invalid JSON as stale state and recreate it instead of crashing the benchmark flow.
- In this repository, any script that launches Playwright Chromium directly should close it in a `finally` block. Otherwise a failed benchmark run can strand headless Chrome processes on the desktop machine.
- In this scene, pre-scaling HDRI and light intensities for half-float-friendly buffer ranges only works if the final exposure applies the exact inverse compensation. Scaling one side without the other will make the whole frame appear black.
- When loading optional PBR channels from object-shaped URL maps, do not rely on `Object.values()` ordering. Build an explicit key-to-texture map first so metallic, normal, and roughness channels cannot silently swap slots.
- In Playwright headless mode, the onscreen FPS counter is not a reliable performance gate because `requestAnimationFrame` can be throttled when the page is not foregrounded. Use an explicit render-cost benchmark that repeatedly renders and calls `gl.finish()` instead.
- The linked 4K fire flipbook atlas carries large transparent margins, so the torch billboard must crop to the measured non-transparent bounds and seat the quad relative to the sconce top instead of assuming the visible flame fills the whole frame.
- The vendored SSR shader in this project was double-counting HDRI contribution through its env-map fallback, so fixing sky/scene overbrightening required disabling that fallback rather than just turning down the exposed SSR intensity slider.
- `realism-effects` `SSGI` is not currently drop-in compatible with this repository's `three` version because it imports `WebGLMultipleRenderTargets`, so AO/GI dropdown options should stay limited to the modes that actually build and run here.
- Copying the camera's world quaternion straight onto a billboard fails under rotated parent groups. The child billboard needs the camera orientation converted into the parent's local space first.
- In this repository's vendored SSR pass, unsupported meshes must be hidden during the g-buffer render. Leaving them visible while merely skipping the MRT material replacement still lets unlit or transparent meshes contaminate SSR.
- The `@react-three/postprocessing` lens flare effect's `opacity` uniform fades the flare out rather than making it stronger, so visible flare strength should primarily come from `colorGain`, `glareSize`, `flareSize`, and `haloScale`.
- Post-effect debug controls are only trustworthy when they map to the actual wrapped effect props. In this repository, Bloom kernel size requires the non-`mipmapBlur` path, and Depth Of Field needs separate `focusDistance`, `focalLength`, and `bokehScale` controls.
- Browser smoke tests for this project become much more stable when they can force a specific persisted maze via `?maze=<id>` instead of relying on a random runtime layout.
- Once `three-atmosphere` is removed from the scene, remove the package itself as well. Leaving it installed makes the repo harder to reason about and can mask whether the runtime still depends on it.
- The benchmark harness must wait for the Playwright web-server port before both smoke and perf runners. They share the same local `serve-root` port, so only guarding the smoke runner leaves a real race.
- In this repository, a `7x7` torch maze can still clear the `>=120 FPS` perf gate if only the nearest torches cast shadows and distant point lights are culled entirely. Paying for every local point-light shadow cube at once is what tanks the benchmark.
- In this repository, default ambient occlusion can cost more than the nearby torch-shadow budget. If the perf gate suddenly halves, check whether a default post effect forced the composer back on before over-optimizing the lights.
- In this repository, sparse keyboard-state maps must be converted with `Boolean(...)` first. Using `Number(keys.current.KeyW)` when the key is absent yields `NaN`, which can corrupt the entire player and camera position on the first frame.
- A non-black clear color is not enough proof that the scene rendered. Smoke brightness checks should sample an area expected to contain scene content, not just a background that can pass while the world is invisible.
- In this repository, "HDRI skybox and flame billboards render, but lit scene meshes do not" is a strong sign that the lit-material shader path is the problem. Check the real maze wall in the beauty pass, and simplify the lit material variant before assuming the textures or mesh placement are wrong.
- In this repository, per-maze baked torch lightmaps are a practical way to keep full PBR wall and ground materials without exploding the forward-renderer shadow sampler count.
- A broad maze screenshot can miss baked wall-light regressions because the floor dominates the sampled pixels. The reliable smoke proof here is a deterministic wall-facing view on a known lit wall.
- The baked wall-face rotation must use the same Y-rotation sign as three.js itself. A hand-written sign error can bake the torch contribution onto the opposite side of z-axis walls while the meshes still render correctly.
- In this repository, post-effect smoke crops must avoid the debug controls panel itself. A centered crop can silently measure checkbox and slider changes instead of the scene and produce false visual failures.
- In this repository, baked torch-light textures behave much more predictably when they store normalized grayscale intensity with headroom instead of clipped warm RGB values. The grayscale bake avoids hue artifacts while still letting the PBR wall and floor materials carry the visible surface detail.
- In this repository, enabling optional post effects should not switch the scene between two different tone-mapping pipelines. Keeping the composer path active for the default scene makes zero-strength effect checks meaningful.
- In this repository, source-controlled baked lightmaps embedded directly into the persisted maze modules can bloat the main JavaScript chunk and materially slow cold startup. If startup becomes a priority again, move baked lightmap payloads out of the main app bundle before chasing smaller rendering tweaks.
- In this repository, recapturing dozens of local reflection probes every time a debug lighting slider changes is too expensive for interactive use and can stall browser automation. Keep the current probe captures static unless a dedicated incremental recapture path is built.
- In this repository, driving `scene.environment` from the camera's current maze cell makes the whole PBR lighting stack appear to flicker as the player moves. Keep the global HDRI as `scene.environment` and assign local probe env maps to the reflective meshes instead.
- In this repository, assigning per-cell local reflection probes to rough maze wall materials creates obvious hard seams and implausible bright exterior faces. Keep local probes on the reflective floor patch and metal sconces, and let the rough stone walls rely on the baked lightmaps plus the global HDRI.
- In this repository, the maze floor patch needs separate UV channels for its world-scale PBR textures and its baked lightmap. Reusing one UV set for both prevents tiling continuity or local-probe tiling from lining up correctly.
- In this repository, smooth local-reflection transitions need probe tiles bounded by probe centers, not tiles centered on probes themselves. That keeps bilinear probe weights continuous across the floor instead of replacing one probe abruptly with another at cell edges.
- In this repository, probe-blend `onBeforeCompile` hooks must stay attached across uniform-only mode toggles. Clearing the hook while reusing the same custom program cache key can leave the React state saying `world` or `none` while the cached shader uniforms stop updating.
- In this repository, custom vertex-shader world-position varyings should not depend on `worldPosition` existing after `#include <worldpos_vertex>`. Some material variants compile without that symbol, so compute the probe world position explicitly before `#include <project_vertex>` instead.
- In this repository, assigning an explicit `material.envMap` bypasses `scene.environmentIntensity`. If a material needs a custom env map source, carry the calibrated HDRI intensity onto that material explicitly or the reflections will not match the visible sky.
- In this repository, adding a second raw `RenderPass` for torch billboards inside `@react-three/postprocessing` can blank the main beauty pass and leave only the billboard scene visible. If the world goes black while unlit flames still render, remove the extra pass before chasing lighting values.
- In this repository, a raw `postprocessing` `Pass` mounted through `@react-three/postprocessing` should not assume the composer will populate custom scene references like `mainScene` in the way a bespoke app pass might expect. When a pass must render the actual R3F scene again, wire `useThree((state) => state.scene)` into the pass explicitly.
- In this repository, a post-opaque billboard composite that rejects pixels by comparing billboard depth against opaque-scene depth must render the billboard layer with `depthWrite` enabled and with the scene background disabled for that render. Otherwise the composite can silently discard every billboard pixel or fill the billboard buffer with sky/background instead of only the translucent sprites.
- In this repository, baked wall-lightmaps on `BoxGeometry` must account for three.js's mirrored local `-Z` face UV orientation. Treating `+Z` and `-Z` as if they share the same horizontal direction will invert one wall face at runtime and produce the exact "near side black, far side bright" failure on torch-lit walls.
- In this repository, local torch-bake occlusion must include the sconce body itself, not just maze walls. If the bake ignores the sconce, the attached wall cannot develop the expected shadow band beneath the torch even when the geometry and textures are otherwise correct.
- In this repository, generated wall lightmaps should not use the default image-texture Y flip on upload. The wall bake writes row `0` as the bottom of the wall, so reusing ordinary texture flip behavior will turn wall-mounted torch shadows upside down at runtime.
- In this repository, `@react-three/postprocessing`'s `SSAO` radius should be mapped near the library's own large default scale. Feeding it tiny sub-unit radii can make the effect look like a no-op even though the pass is technically active.
- In this repository, the wrapped `LensFlare` component from `@react-three/postprocessing` is not a safe drop-in for the torch scene. Its prop surface collides with blend opacity semantics and the runtime path was unstable enough to hang headless checks, so a small in-house flare overlay is safer than trying to tune that third-party effect blindly.
- In this repository, baked torch-wall occlusion must match the actual sconce shell. Treating the sconce as a full sphere falsely blocks rays through the fixture's open upper half and creates the exact "lower half of the wall is black" regression.
- In this repository, multiple `LensFlareEffect` instances from `@react-three/postprocessing` collide in the merged effect shader because the upstream fragment shader redeclares `vTexCoord` globally. Without patching that upstream shader, only one real lens-flare effect instance can be active at a time.
- In this repository, a persisted-maze lightmap format bump should rebake the existing maze modules in place rather than regenerating maze topology. Full topology regeneration turns a quick asset rewrite into a many-minute rebuild and can silently replace authored mazes.
- In this repository, local reflection-probe captures must wait until the opaque maze floor patch, wall materials, and sconces are mounted with their real captureable materials. A one-shot bake that runs too early can produce probes containing little more than the HDRI and temporary torch emitters.
- In this repository, `@react-three/postprocessing` lens flares expect `screenRes` in canvas pixels, not the React Three Fiber world-space viewport size. Feeding world units into that uniform can make the flare appear broken even when the effect is otherwise enabled.
- In this repository, `@react-three/postprocessing`'s lens-flare shader already adds `inputColor` internally. Keep its composer blend mode at `NORMAL`; switching the wrapper effect to additive blending double-composites the scene and blows the whole frame out.
- In this repository, the lens-flare shader's `colorGain` uniform behaves like a normal RGB multiplier, not a large arbitrary intensity scalar. Values near `0..1` are sane; values in the tens will wash out the frame.
- In this repository, a one-shot reflection-probe bake must wait for every expected ground patch, wall mesh, and sconce mesh to be ready, not merely for any one representative mesh to appear ready. Otherwise the baked probes can permanently miss maze occluders and show torches through walls.
- In this repository, the one-minute smoke budget is tight enough that the harness itself must be profiled like production code. Record the slowest smoke phases in `logs/latest-smoke-profile.json`, keep screenshot count minimal, and replace expensive browser actions with cheaper state assertions when they prove the same behavior.
- In this repository, once startup alone grew close to the one-minute budget, the benchmarked smoke runner had to stay narrowly focused on startup readiness and a few live shader/control assertions. Slower end-to-end render checks belong in a separate Playwright integration spec so performance enforcement stays honest without dropping coverage.
- In this repository, the Playwright `*:runner` commands serve the root-published bundle. If `src/` changes are not followed by `npm run build:pages`, those runners can silently validate stale `assets/bootstrapApp.js` and give false confidence.
- In this repository, parallel Playwright invocations that share `serve-root` on `127.0.0.1:42731` are invalid measurements. Run them sequentially or give them distinct ports.
- In this repository, reflection-probe startup improved materially once the app baked the spawn-area probes first and deferred the rest to lower-priority follow-up work instead of blocking readiness on the entire maze.
- In this repository, do not dispose assets returned directly from `useLoader(...)` in component cleanup. React Strict Mode remounts in Vite development can reuse the cached loader result on the second mount, and disposing the shared HDR or texture object in the first cleanup can leave the dev scene stuck behind the loading overlay even though production builds still work.
- In this repository, a reflection-probe debug view should display the actual HDR cubemap used for local reflections, not a separate low-resolution 8-bit proxy. The low-dynamic-range proxy can clip warm torch colors to white and erase wall/floor detail even while the real probe path is fine.
- In this repository, probe correctness must be verified on the capture payload itself, not inferred from a UI toggle or a separate preview path. A cheap cubemap-detail metric exposed through the debug state is more reliable than repeatedly screenshotting probe spheres and is fast enough to stay under the smoke-test budget.
- In this repository, the reflection-probe debug spheres are for human inspection of the actual processed local reflection source used by runtime shading. Keep raw cubemap capture debugging available as a separate exported or on-demand atlas instead of silently swapping the sphere preview to a different data source.
- In this repository, `CubeCamera` probe captures only render the layers enabled on the cube camera itself. If torch billboards live on a dedicated layer, the probe capture must explicitly enable that layer or the debug spheres will misleadingly omit the real flames even while the rest of the maze geometry is captured.
- In this repository, reflection-probe debug spheres should not reuse gameplay exposure. A fixed neutral preview tone map keeps bright captured emitters readable and prevents exposure-dependent false negatives while diagnosing probe contents.
- In this repository, raw reflection-probe debugging and automated readback must be validated separately from the capture scene itself. Our half-float raw-probe path plus `readRenderTargetPixels(..., Uint8Array)` produced misleading evidence, while an `UnsignedByteType` raw cube target gave consistent capture, preview, and smoke-test results. If probe walls seem to disappear again, compare target types and debug instrumentation before blaming the maze materials or the capture timing.
- In this repository, the most trustworthy automated proof of raw reflection-probe contents is an on-demand atlas rendered from the exact raw cubemap faces, not inference from a tiny sphere preview and not unsupported half-float readback. Use the debug atlas path when the human and the probe metrics disagree.
- In this repository, clearing a saved `onBeforeCompile` shader reference during React effect cleanup can silently break later uniform-only updates even while the material keeps reusing the same compiled program. If probe-blend state changes but the rendered result does not, inspect the actual bound shader uniforms before assuming the local reflections are active.
- In this repository, `vite.config` bakes the Git revision into the built asset from the current `HEAD`. Rebuilding before committing leaves the deployed overlay claiming the previous commit even when the code changes are present, which makes visual bug reports much harder to reconcile.
- In this repository, a geometry-only reflection-probe atlas rendered with `scene.overrideMaterial = new MeshBasicMaterial({ color: 'white' })` against a black background is the cleanest automated splitter between capture-stage failures and later PMREM or material-stage failures. Use that before arguing from beauty-pass screenshots.
- In this repository, local probe PMREM textures cannot be sampled through `textureCubeUV` with shader constants compiled from a differently sized global env-map PMREM. If the global HDRI PMREM and local probe PMREMs differ in CubeUV size, local reflections will read the wrong layout and appear broken even when cube-camera capture itself is correct.
- In this repository, geometry-only capture proof is not enough when the suspected bug is "the real material fails to render in the capture." The correct next step is the same method that worked for the sconce: isolate the real pass, start from a known-good material, and add the actual material features one by one until the first failing step is reproduced.
- In this repository, when the human says a mesh is plainly not drawing, do not reinterpret that as a perception problem without proof. Use the exact failing render path and falsifiable artifacts to determine whether the material, not the geometry, is the first thing that breaks.
- In this repository, an open-top sealed probe-occlusion maze can produce a false positive because the center probe still sees the surrounding wall caps, and the probe-only torch lights can legitimately light those visible top faces. Use that layout to separate bake-vs-capture issues, but do not treat it as proof that torches are visible through the vertical wall faces.
- In this repository, a real-maze raw probe atlas and an isolated raw-probe preview are stronger proof than a broad in-scene impression. If the raw atlas and the isolated preview both show walls, the next question is whether the preview context or interpretation is misleading, not whether the cube camera omitted the maze geometry.
- In this repository, when the question is "is the runtime actually using the baked probe texture or some other reflection source," prove it by comparing all three of these in one session: the baked probe UUID from `getReflectionProbeState()`, the bound material/probe-sphere UUIDs, and the runtime raw/processed atlas dumps. If those all match the exported artifact PNGs, the bug is downstream of capture and texture binding.
- In this repository, do not call `currentProgram.getUniforms().setValue(...)` for built-in materials from `material.onBeforeRender`. That mutates three.js's uniform cache before the correct program is bound, which can leave the live GL uniform stale while the cache incorrectly claims the value was uploaded.
- In this repository, the principled way to extend built-in PBR materials is through three.js's supported hooks: `customProgramCacheKey`, `onBeforeCompile`, and uniform updates on the patched shader object itself. Do not try to synchronize built-in-material custom uniforms by mutating renderer internals like `gl.properties.get(material).uniforms`.
- On this Windows/Node 18 setup, `node:test` child-process wall time can include large teardown noise that is not reflected in TAP `duration_ms`. For this repository's unit-test budget tracking, log both the reported TAP durations and the outer wall time, and enforce the budget on the reported test durations rather than the child-process teardown artifact.
- In this repository, source-controlled maze modules should not be imported directly into the browser runtime when they carry large baked-lightmap payloads. Keep the synchronous maze registry for Node tests and scripts, but use a browser-only lazy loader for the runtime so the initial app bundle stays small.
- In this repository, turning off baked lightmaps or turning on probe IBL must also disable the baked ambient-lightmap channel on affected materials. Zeroing only the torch lightmap intensity leaves a hidden ambient term active and makes the debug toggles appear broken.
- In this repository, a full-resolution CPU sky-ambient bake is too slow for practical persisted-maze rewrites. Bake the high-frequency torch term at full resolution, but sample the low-frequency ambient term on a coarse grid and upscale it smoothly into the atlas.
- In this repository, the startup benchmark should align with the real requirement: immediate shell first, scene later. Waiting on the browser `load` event hides the improvement from an inline bootstrap overlay and lazy maze loading.
- In this repository, horizontal world-space fog slices are a bad fit for first-person inspection because they can become nearly invisible at eye level even while their uniforms update correctly. A depth-aware screen-space fog pass is a more reliable implementation path for the requested full-volume haze.
- In this repository, post-effect tests should only claim a visual proof when the frame-delta assertion is actually stable in the authored scene. Wiring a canonical pass into the composer is not, by itself, proof that the scene now shows a useful effect.
- In this repository, persisted CubeUV probe PNGs need to be loaded with image bitmap Y orientation matching the exported texture layout. If processed probe debug spheres show black or unrelated faces while raw cubemap dumps look good, inspect the persisted CubeUV image loader options before changing probe capture math.
- In this repository, baked HDR lightmap artifacts need two distinct exports: a preview PNG for quick inspection and a raw HDR export for debugging the real payload. Reusing the preview path for both can make the artifact look banded or clipped even when the underlying float data is smooth.
- In this repository, a debug camera setter used by smoke tests must explicitly switch the camera out of turn-based player-follow mode. Otherwise the turn controller can restore the gameplay camera on the next frame and make material-uniform assertions inspect the wrong part of the maze.
- In this repository, first-order volumetric-lightmap SH directionality is cheap enough to unit-test directly with synthetic cubemap faces. Use that deterministic test to catch face-axis swaps instead of relying only on hard-to-read in-scene probe spheres.
- In this repository, the 4K fire flipbook is too large to be part of Suspense-gated first scene readiness. Load it asynchronously after the maze scene mounts so the loading overlay can meet the sub-5-second fade requirement without dropping the final billboard asset.
- In this repository, rendering multiple separate upstream `LensFlareEffect` passes can make the debug scene run at only a few FPS. Prefer one stable selected visible flare until a shared multi-light flare implementation exists.
- In this repository, sampler-pressure fixes for the probe shader should be as narrow as possible. Compiling out the extra local-radiance probe samplers is safe; disturbing the shared diffuse and volumetric-lightmap plumbing is easy to break and should be avoided unless a test proves the new variant end to end.
- In this repository, never put debug-state object rebuilding on a material's `onBeforeRender` hot path. The probe-blend materials render often enough that per-draw `map`/`clone`/object-allocation debug work can dominate frame time, and any extra scene pass like AO will multiply that cost immediately.
- In this repository, the Vite React Fast Refresh transform can make the development server effectively unusable once `App.tsx` becomes this large. If `localhost` wedges while the built app remains responsive, disable Fast Refresh in `vite.config.ts` before chasing renderer bugs.
- In this repository, extracting `rgb16f` lightmap sub-rects by decoding every half-float channel to JS numbers and re-encoding them back to half-floats is too expensive for startup. Copy the original half-float row data directly into the face texture whenever the runtime atlas is already `rgb16f`.
- In this repository, `HalfFloatType` runtime lightmaps must use `RGBAFormat`, not `RGBFormat`. On ANGLE/D3D11, `RGB + HalfFloat` can fail texture allocation with `GL_INVALID_ENUM`, which collapses baked wall lighting and can stall the whole page under a flood of WebGL errors.
- In this repository, runtime volumetric-lightmap occlusion should be connectivity-gated by maze cells, not by persisted probe-depth atlases. The depth-atlas path created extra artifact churn and sampler pressure; use per-cell wall connectivity for diffuse/fog rejection unless a future feature proves it needs true depth visibility.
- An alternate Vite port can be useful as a temporary diagnostic when startup behaves differently across ports, but it should not become a permanent redirect or replace the repo default. Keep the default dev URL on `127.0.0.1:5173`.
- In this repository, a maze regeneration test can hang if a test factory repeatedly returns a maze that passes structural validation but fails timing validation. Bound regeneration attempts and normalize timing metadata in fixtures so invalid factories fail with a useful diagnostic instead of looping forever.
- In this repository, offline reflection-probe export must not depend on the runtime environment-light texture, because gameplay intentionally clears runtime global environment lighting. Probe export should use the scene's HDRI background as its capture environment when the runtime environment texture is null.
- In this repository, full persisted reflection-probe export is an offline asset build step, not a fast validation step. Keep it measured separately from maze generation/validation so the 5-second maze budget is not conflated with a multi-minute 245-probe asset export.
- In this repository, loading every persisted reflection probe into runtime GPU memory can pass early startup tests while still freezing the real page minutes later. A perf gate must wait for probe residency to stabilize and assert resident count, texture-memory budget, and default-scene frame time.
- In this repository, low triangle count does not guarantee low render cost. The AWIL werewolf source asset was only about 7.8k triangles but shipped as 163 static skinned meshes, costing roughly 25ms/frame by itself at 1440p until it was converted offline into a single regular runtime mesh.
- In this repository, a steady-state render benchmark can miss startup freezes caused by first-use WebGL work. Track live `requestAnimationFrame` gaps from navigation through lightmap, probe, and postprocessing warmup when diagnosing reports that the page freezes as assets finish loading.
- In this repository, 16-bit source PNG material textures are inappropriate browser startup assets even when the authored source resolution is correct. If the authored 4K fire flipbook must be used directly, keep eager preload tags away from it and defer its decode until after the loading overlay is responsive.
- In this repository, first gameplay frames must not be responsible for shader compilation, texture upload, or postprocessing composer initialization. Warm scene textures, representative mesh renders, and composer passes before the loading overlay fades.
- In this repository, static-camera render-cost checks are not enough for playability. The solution-replay path can exercise monsters, camera motion, and probe/lightmap sampling patterns that a fixed camera misses, so keep it in the performance suite.
- In this repository, startup warmup work must yield aggressively. Rendering many warmup objects in one RAF can create the exact loading-overlay stutter the startup gate is designed to catch, even if steady-state frame time is fine.
- In this repository, a lit mesh that writes depth but no scene color is a strong signal that the fragment shader failed to link or exceeded WebGL sampler limits. Probe/lightmap shader chunks must compile out unused radiance, coefficient, and depth-atlas samplers instead of merely setting their runtime intensities to zero.
- In this repository, when runtime material sampler requirements are reduced, update every readiness predicate that waits for material texture slots. A stale capture-readiness check that still requires removed maps can block loading or probe export indefinitely even though the rendered materials are correct.
- In this repository, material shader debug state must not be marked as applied before `onBeforeCompile` has produced a shader object. If a material key changes while a mesh is off-camera, `onBeforeRender` needs to perform the first real uniform upload after compilation instead of skipping it because React state already ran.
- In this repository, level data residency and rendered level residency must stay separate. Background preloading two-hop transition neighborhoods is useful only if the renderer still limits the live scene to the current level plus its immediate visible neighbors.
